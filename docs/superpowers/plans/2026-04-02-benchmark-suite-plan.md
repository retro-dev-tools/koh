# Benchmark Suite Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement a head-to-head KOH vs RGBDS benchmark (Docker-based, Bash orchestration + Python data/report generation) and BenchmarkDotNet microbenchmarks for KOH's internal pipeline phases.

**Architecture:** Two independent components: (1) Bash shell scripts + Python helpers + Dockerfile for head-to-head CLI benchmarking against pokecrystal, orchestrated via `make benchmark`; (2) a BenchmarkDotNet .NET project for KOH-internal microbenchmarks using pokecrystal files as input.

**Tech Stack:** Bash, Python 3, Docker, GNU time, .NET 10, BenchmarkDotNet 0.15.8, RGBDS 1.0.1

**Spec:** `docs/superpowers/specs/2026-04-02-benchmark-suite-design.md`

**Key design decisions:**
- **ALL structured data** (JSON, markdown) is generated by Python — Bash only captures raw key-value pairs and exit codes into flat text files, never assembles JSON
- KOH CLI binaries published to separate subdirectories to avoid output collisions
- BenchmarkDotNet 0.15.8 is the chosen version — but the actual runtime enum names and exporter symbols MUST be verified against the installed package after `dotnet restore`, not assumed from documentation
- C# benchmark code in this plan is **placeholder pseudocode** — the implementing agent must inspect actual KOH API signatures and mirror existing production/test patterns, not copy these snippets verbatim

### Task 3: Python report generator and data helpers

This is the central data-processing component. ALL structured data (JSON, markdown) flows through this script. Bash never assembles JSON.

**Files:**
- Create: `benchmarks/generate-reports.py`

The script reads:
- `$BENCH_OUTPUT/run-data/run-N.txt` — one flat text file per run with key=value lines per phase
- `$BENCH_OUTPUT/environment.txt` — captured environment key=value pairs
- `$BENCH_OUTPUT/preflight.txt` — pre-flight classification key=value pairs
- `$BENCH_OUTPUT/rom-data.txt` — ROM comparison key=value data (if any)

It produces:
- `$BENCH_OUTPUT/preflight.json`
- `$BENCH_OUTPUT/raw.json`
- `$BENCH_OUTPUT/results.md`
- `$BENCH_OUTPUT/rom-comparison.json`

- [ ] **Step 1: Create generate-reports.py**

The agent should implement this script with the following design:

**Input parsing:** read flat key=value text files. Each run file has lines like:
```
rgbds_assembly.status=ok
rgbds_assembly.wall_s=1.25
rgbds_assembly.user_cpu_s=1.10
rgbds_assembly.sys_cpu_s=0.10
rgbds_assembly.peak_rss_kb=43008
rgbds_assembly.exit_code=0
rgbds_assembly.log_path=logs/run-1-rgbds-assembly.log
rgbds_assembly.phase_command=rgbasm -Q8 -P includes.asm -Weverything -Wtruncation=1 -o ${asm}.o ${asm}.asm
rgbds_assembly.files=audio,home,main,ram,...
koh_assembly.status=unsupported
koh_assembly.classification=unsupported_cli_feature
koh_assembly.reason=missing -P -Q CLI flags
```

**Aggregate computation:**
- For each phase, classify aggregate status:
  - `ok` — all planned runs succeeded
  - `partial` — some runs ok, some failed/skipped
  - `unsupported` — phase was unsupported (never attempted)
  - `no_data` — phase was supported but no runs succeeded
- Compute medians only from `ok` runs
- Track `success_count`, `executed_count`, `planned_count` separately
- Speedup shown only when both tools have `success_count == planned_count`

**Markdown rendering:**
- Render `partial` and `failed` states distinctly from `unsupported`
- For ROM mismatch: include differing SHA256 values, first differing offset, reason
- For ROM postprocess failure: include reason and tool that failed

**ROM comparison JSON:**
- Store `rgbfix_command_rgbds` and `rgbfix_command_koh` separately
- Include `rgbds_header_hex`, `koh_header_hex` when available

**Phase command recording in raw.json:**
- Each executed phase entry must include:
  - `measurement_tool`: `/usr/bin/time -v`
  - `phase_command`: structured object with `per_file_template` + `files` for assembly, or `argv` for linking

### Task 4: Container-side harness — run-benchmark.sh

Bash harness that orchestrates benchmark execution. All structured output is delegated to the Python generator. Bash only writes flat key=value text files — never JSON.

**Files:**
- Create: `benchmarks/run-benchmark.sh`

- [ ] **Step 1: Create run-benchmark.sh**

The agent should implement this script following these rules:

**No JSON in Bash.** Every output is a flat text file with `key=value` lines. Python handles all JSON/markdown.

**All `local` usage inside functions only.** The main script body uses only global variables.

**Structure:**

1. **Configuration** — read env vars with defaults. Define `ASM_FILES` array, capability matrix arrays.

2. **Functions:**
   - `log()`, `die()` — logging
   - `has_feature()` — check if a feature is in a list (uses `local`)
   - `all_required_present()` — check all required features are supported (uses `local`)
   - `clean_outputs()` — remove `.o`, `.koh.o`, `.gbc` files (uses `local`)
   - `run_rgbds_assembly()`, `run_koh_assembly()`, `run_rgbds_linking()`, `run_koh_linking()` — phase execution functions wrapping `/usr/bin/time -v` (use `local`)
   - `parse_time_output()` — parse `/usr/bin/time -v` log, write key=value pairs to a file (uses `local`)

3. **Step 1: Record environment** — write key=value pairs to `$BENCH_OUTPUT/environment.txt`:
   ```
   cpu_model=AMD Ryzen 7 5800X
   kernel=6.1.0
   container_os=Ubuntu 24.04.1 LTS
   ram_mb=32768
   dotnet_runtime=10.0.0
   dotnet_sdk_host=10.0.100
   rgbds_version=1.0.1
   koh_version=0.1.0
   ```

4. **Step 2: Clone pokecrystal** at pinned `POKECRYSTAL_REF`.

5. **Step 3: Build prerequisites** — inspect pokecrystal's Makefile to determine exact asset-generation targets. At minimum run `make -C tools/` then `make` for asset generation. Do NOT use `|| true` — errors should propagate. After build, validate that representative generated assets exist (e.g., check a few `.2bpp` files referenced by `INCBIN` in the source).

6. **Step 4: Pre-flight classification** — write key=value pairs to `$BENCH_OUTPUT/preflight.txt`:
   ```
   koh_asm_supported=false
   koh_asm_classification=unsupported_cli_feature
   koh_asm_reason=missing -P -Q CLI flags
   koh_asm_required=-P,-Q
   koh_asm_have=
   koh_link_supported=false
   koh_link_classification=unsupported_benchmark
   koh_link_reason=missing -l (linker script) support
   koh_link_required=-l
   koh_link_have=
   ```
   Also capture `--help` output to log files for auditing.

7. **Step 5: Warmup** — for each warmup iteration, mirror measured execution:
   - Clean outputs
   - Run RGBDS assembly, then RGBDS linking (using assembly outputs)
   - If KOH assembly supported: clean KOH outputs, run KOH assembly, then KOH linking if supported (using KOH assembly outputs)
   - Same phase dependency chain as measured runs

8. **Step 6: Measured runs** — for each run:
   - Clean outputs
   - Execute each supported phase, capture timing to log files
   - After each phase, call `parse_time_output` to extract metrics from the time log
   - Write one flat text file `$BENCH_OUTPUT/run-data/run-N.txt` with all phase results as key=value lines
   - **Skipped logic:** if KOH linking is supported but KOH assembly failed or was unsupported for this run, write `koh_linking.status=skipped` with reason
   - **Assembly loop failure:** use `set -e` inside the timed subshell so the first assembler error terminates the phase immediately
   - Track executed/success counts as simple counter variables

9. **Step 7: ROM comparison** — run a dedicated post-benchmark validation build (not relying on leftover state from measured runs):
   - Clean all outputs
   - Run RGBDS assembly + linking fresh
   - If KOH phases supported: run KOH assembly + linking fresh
   - Apply `rgbfix` using an array (not string splitting): `RGBFIX_CMD=(rgbfix -Cjv -t PM_CRYSTAL ...)`; `"${RGBFIX_CMD[@]}" pokecrystal.gbc`
   - Write results to `$BENCH_OUTPUT/rom-data.txt` as key=value pairs:
     ```
     status=match|mismatch|skip|rom_postprocess_failure
     reason=...
     rgbds_sha256=...
     koh_sha256=...
     rgbds_size=...
     koh_size=...
     first_diff_offset=...
     rgbds_header_hex=...
     koh_header_hex=...
     rgbfix_command=rgbfix -Cjv -t PM_CRYSTAL ...
     rgbfix_rgbds_exit=0
     rgbfix_koh_exit=0
     rgbfix_rgbds_log=logs/rgbfix-rgbds.log
     rgbfix_koh_log=logs/rgbfix-koh.log
     ```

10. **Step 8: Generate reports** — call `python3 /usr/local/bin/generate-reports.py`

11. **Exit code logic** — as specified in the design spec.

**Key constraints the agent must follow:**
- No `local` outside functions
- No `bc` (use `awk` for arithmetic)
- No inline JSON assembly
- `rgbfix` command stored as array, not string
- Log files: one combined file per phase/run containing stdout + stderr + time output
- Unsupported phases produce no logs — their classification is in `preflight.txt`
