# Game Boy 2048 Sample — Design

A complete, polished, GBC-only `2048.gbc` ROM that ships under `samples/gb-2048/` as a flagship Game Boy showcase for the Koh toolchain. Companion to `samples/KohUI.Demo/`: KohUI.Demo shows off the C# UI framework strand of the project; this sample shows off the assembler/linker/LSP strand on a real, playable, non-trivial game.

## Goals, in priority order

1. **Toolchain showcase, not a hello world.** Exercise multi-bank ROMs, the linker, far calls, `INCLUDE`/`INCBIN`, macros, GBC-only hardware features, and SRAM saves. A reader skimming the source should walk away convinced Koh handles a real Game Boy project, not just toy fixtures.
2. **Polished gameplay.** Smooth 8-frame slide animation, merge pop, score + persistent best score, win sequence on first 2048, game-over screen, restart. Feels like a real game, not a tech demo.
3. **GBC-native visuals.** Each tile value gets its own background palette so 2048 reads at a glance the way the canonical web version does.
4. **Buildable from CI** as a separate target so toolchain regressions that break a real project are caught.
5. **LSP-friendly source layout** — a `koh.yaml` so opening `samples/gb-2048/` in VS Code with the Koh extension lights up diagnostics, navigation, and hover.

The first item is the tiebreaker for design choices throughout. Where two approaches both work but one exercises more of Koh, prefer the one that exercises more of Koh.

## Non-goals (YAGNI)

- DMG support — single-target GBC. CGB flag = `$C0`.
- Two-player link cable.
- Music. Sfx + a single wave-channel jingle is the audio budget.
- Undo, replay, hint AI, theme switcher.
- Animated title-screen logo. A static logo with a STAT-driven gradient is enough.
- RGBDS `.o` output. `koh-asm`'s native `.kobj` only.

## ROM banking (MBC5)

Four banks, deliberately split even though a 32 KiB build would fit, to demonstrate the linker:

| Bank | Contents |
|------|----------|
| ROM0 | engine + IRQs + dispatcher, OAM-DMA trampoline source, `farcall` helper |
| ROM1 | game logic — rng, board, score, save, anim, render |
| ROM2 | graphics — tile data + font, behind a small loader |
| ROM3 | screens — title, win, gameover |

A `farcall <bank>, <symbol>` macro saves the current bank, switches via `[$2000]` write, calls, restores. Used wherever ROM0 dispatches into ROM1/ROM2/ROM3.

## Memory map

### WRAM0 (`$C000–$CFFF`)

| Symbol | Size | Purpose |
|---|---|---|
| `wBoard` | 16 | Current board, exponent encoding (0=empty, 1=2, 2=4, …, 11=2048, 12=4096) |
| `wPrevBoard` | 16 | Snapshot before the current move; basis for animation |
| `wMoveIntents` | 64 | Per source cell: `{old_idx:1, new_idx:1, value:1, flags:1}` × 16 |
| `wScore` | 4 | 32-bit score, little-endian |
| `wBestScore` | 4 | 32-bit best score |
| `wRngState` | 2 | 16-bit Galois LFSR state |
| `wInput` | 4 | Pad: current, previous, edge, repeat counter |
| `wGameState` | 1 | TITLE / PLAYING / ANIMATING / WIN / GAMEOVER |
| `wWonOnce` | 1 | Set after first 2048 reached so WIN fires once per run |
| `wAnimFrame` | 1 | 0–11 (8 slide + 4 pop) |
| `wOAMBuffer` | 160 | Shadow OAM, aligned at `$C100` (placed via `SECTION "OAM Buffer", WRAM0[$C100]`) |
| `wVBlankQueue` | 64 | Pending tile-write commands; format below |

VBlank queue entry: `[cmd:1][dst_lo:1][dst_hi:1][len:1][data:N]`, terminator = cmd `$00`. Drain in IRQ until terminator or budget exceeded.

### HRAM (`$FF80–$FFFE`)

`hOAMDMA` (8 bytes) — runtime-resident OAM-DMA trampoline copied from ROM0 at boot. `hFrameCounter`, `hSysFlags`, scratch.

### SRAM (`$A000–$BFFF`, MBC5 bank 0, 8 KiB)

| Offset | Size | Field |
|---|---|---|
| `+0` | 4 | Magic `"K248"` |
| `+4` | 4 | Best score |
| `+8` | 16 | Last-played board (exponents) |
| `+24` | 4 | Last-played score |
| `+28` | 2 | Fletcher-16 checksum over `+0 .. +27` |

Magic and checksum gate save loading — anything else and we treat SRAM as uninitialised, which is also the first-boot path.

## Game state machine

```
TITLE ──START──► PLAYING ──D-pad──► ANIMATING ──┐
   ▲                ▲                           │
   │                │             (commit, spawn, run win/lose checks)
   │                │                           │
   │                │             ┌─── no win, no lose ──► PLAYING
   │                │             │
   │                │             ├─── 2048 reached and !wWonOnce ──► WIN ──A──► PLAYING (wWonOnce=1)
   │                │             │
   │                └─────────────┴─── board full + no equal adjacents ──► GAMEOVER ──START──► TITLE
   │
   └─ (no path)
```

Win and lose checks fire at the *end* of ANIMATING (after the new tile spawns), never mid-slide. After WIN, `wWonOnce` is set so play can continue past 2048 toward 4096 without retriggering.

- **TITLE** — STAT-split background (gold gradient top, board preview bottom), animated value carousel (palette cycle), "PRESS START". START → PLAYING; SELECT (held 1 s) → wipe SRAM after confirmation prompt.
- **PLAYING** — poll input, dispatch on D-pad edge.
- **ANIMATING** — 8-frame slide (2 px/frame), then 4-frame merge pop on merged cells, then commit and spawn.
- **WIN** — fires once per run on first 2048, plays wave-channel jingle, "YOU WIN" banner, A returns to PLAYING and play continues toward 4096.
- **GAMEOVER** — dim board palette, banner, score + best, START → TITLE.

## Move logic

Encode cells as **exponents** (1=2, 2=4, …, 11=2048). One canonical `MoveLeft` routine:

1. Compact left (skip zeros).
2. Single forward pass: if `r[i] == r[i+1]` and *neither cell* has been merged in this pass, set `r[i] += 1`, `r[i+1] = 0`, `score += 2^r[i]`, mark `r[i]` merged, advance `i++`. The "already-merged" guard is what makes `[4 4 4 0]` left → `[8 4 0 0]` (one merge) instead of `[16 0 0 0]` (would-be cascade).
3. Compact left again.

Other directions reuse `MoveLeft` via four 16-byte index transposition tables (left/right/up/down). While processing, fill `wMoveIntents` with the `(source-cell → destination-cell, value, merged?)` tuples that animation needs.

A move is "valid" iff at least one cell moved or merged. Only valid moves spawn a new tile and advance state. New-tile value = 2 (90%) or 4 (10%); position = `LFSR % count_of_empty_cells`, mapped to actual empty cell by counting.

Win check after spawn: any cell == 11 and `!wWonOnce` → WIN.
Lose check after spawn: board full AND no equal adjacent pair in any row or column → GAMEOVER.

## Animation

Animation consumes `wMoveIntents` over 12 frames:

- **Frames 0–7 (slide).** For each active intent, write 2 OAM entries (`OBJ16` mode, 8×16 each = 16×16 cell) at `lerp(old_pixel, new_pixel, frame/8)`. BG cells whose source matches a slide intent are blanked (palette swapped to background-color tile) so the OAM ghosts aren't doubled.
- **Frame 8.** Snap. Clear ghost OAM. HDMA the new tilemap row range into VRAM.
- **Frames 9–11 (merge pop).** Merged cells swap tile to a "popped" variant for one frame, then to a slightly-larger variant for one frame, then back. New-spawn cell fades in via 2-frame palette ramp.

OAM budget: 16×16 cells in `OBJ16` mode = 2 OAM entries each. Worst case 16 moving cells × 2 = 32 entries, leaving 8 for HUD/decorative sprites (within the 40-slot OAM and 10-per-line caps).

## Rendering

- **Tile geometry:** cells 16×16, gap 4 px, board = 4·16 + 3·4 = 76 px. Origin x = 42, y = 32. HUD on window layer pinned to top 16 px.
- **VRAM (`LCDC.4 = 1`, `$8000` addressing for BG):** BG and OBJ share the tile pool, which means value glyphs are stored once and reused as both committed board cells and sliding ghosts.
  - `$8000–$8FFF` (VRAM bank 0) — Tile pool: 11 value glyphs (2…2048) × 4 8×8 sub-tiles, plus 4096 glyph, plus "popped"/"large" variants used during merge pop, plus 8×8 font glyphs (digits + a few letters), plus a blank/background-color tile. ~200 of 256 slots used; ~56 spare.
  - `$8000–$8FFF` (VRAM bank 1) — Available for additional tile data if `$56` slots run out. Currently unused; reserved.
  - `$9800–$9BFF` (bank 0) — BG tilemap (board + decorations).
  - `$9800–$9BFF` (bank 1) — BG attribute map: per-cell palette index, h/v flip. **This is how each value family gets its own color** — same tile, different attribute palette.
  - `$9C00–$9FFF` (bank 0) — Window tilemap (HUD).
  - `$9C00–$9FFF` (bank 1) — Window attribute map.
- **Palettes (GBC).** 8 BG palettes — one per value family (2/4 share, 8/16 share, etc., grouped to fit in 8 slots) plus one "HUD/font" palette and one "background/decoration" palette. 8 OBJ palettes mirror the BG palette ramp for ghosts so a sliding cell looks identical to its committed BG counterpart. Title screen uses two palette sets (gradient top / board preview bottom) swapped at the band boundary by STAT.
- **All VRAM writes go through `wVBlankQueue`.** Outside VBlank, only OAM (via DMA) and palette writes (via BCPS/BCPD/OCPS/OCPD with auto-increment) touch the GPU side. The boot path turns the LCD off, fills initial VRAM and palettes, then turns it back on — that's the only LCD-off period after reset.

## Hardware features deployed

- OAM DMA every VBlank via the HRAM trampoline.
- HDMA general-purpose mode (CGB GP-DMA, `HDMA5` bit 7 = 0) for board redraw on move-commit and for title-screen tile load.
- Double-speed mode entered after boot (KEY1 prepare + STOP + verify).
- STAT interrupt on title screen for two-band palette split (`LYC` set to gradient/board boundary).
- Window layer pinned at top for HUD.
- Sound: channel 1 = move sfx, channel 2 = merge sfx, channel 3 (wave) = win jingle, channel 4 = game-over noise burst. All sound *data* (note tables, jingle steps, wave-RAM samples) lives in `engine/sound.asm`; screen modules call into the engine by ID (`PlaySfx A=<id>`).

## ROM header

| Field | Value |
|---|---|
| Title | `"KOH 2048"` (padded to 11 bytes) |
| CGB flag (`$0143`) | `$C0` (GBC-only) |
| New licensee (`$0144–$0145`) | `"00"` |
| SGB flag (`$0146`) | `$00` |
| MBC type (`$0147`) | `$1B` (MBC5 + RAM + BATTERY) |
| ROM size (`$0148`) | `$01` (64 KiB, 4 banks) |
| RAM size (`$0149`) | `$02` (8 KiB) |
| Destination (`$014A`) | `$01` (non-Japan) |
| Old licensee (`$014B`) | `$33` (use new licensee) |
| Version (`$014C`) | `$00` |
| Header checksum (`$014D`) | computed |
| Global checksum (`$014E–$014F`) | computed |

**Investigation item.** Verify whether `koh-link` writes the header checksum and complement bytes during link, or whether the sample needs a post-link fixup. The build script branches on this finding. Quick test: link a minimal stub, check `$014D`–`$014F` against the spec formula. Fall-back plan: small in-tree fixup script (PowerShell + bash) under `samples/gb-2048/scripts/` that runs after `koh-link`.

## Build & test integration

- `samples/gb-2048/koh.yaml` declares the project so the LSP indexes it on workspace open.
- `build.proj` adds a `BuildSample2048` target:
  - depends on `PublishDev`-style published `koh-asm` and `koh-link`,
  - assembles every `.asm` under `src/` to `.kobj`,
  - links to `samples/gb-2048/build/2048.gbc`,
  - succeeds iff exit 0 + `2048.gbc` exists + `$0143 == $C0`.
- CI: separate job, **not** wired into the default `Test` target so test runs stay fast. Failure of `BuildSample2048` blocks merge.
- README documents BGB / SameBoy / mGBA as recommended emulators and lists controls.
- A bot-readable `samples/gb-2048/build/.gitignore` keeps the `.gbc` out of the tree.

Optional follow-up (not v0.1): a headless `Koh.Emulator.Core` test under `tests/Koh.Samples.Tests/` that boots the ROM, scripts a few moves, and asserts on `wScore`/`wBoard` state. Useful regression net once the emulator is mature; deferred so the sample doesn't block on it.

## Manual test plan

Run in BGB and verify:

- Title screen renders with the STAT palette split intact (no flicker on top/bottom band boundary).
- A first-boot SRAM (or after wiping) shows best score = 0.
- D-pad on PLAYING produces a smooth slide; merges pop and score increases by `2^new_value`.
- Combo merges in a single row (e.g., `[2 2 2 2]` → left → `[4 4 0 0]`, score +8) score correctly — single-pass merge rule, each cell merges at most once.
- 2048 reached → win banner and jingle. A continues play. 4096 does not retrigger.
- Board full with no equal adjacents → GAMEOVER. START → TITLE.
- Reset emulator (SRAM-preserved): best score persists. Wipe SRAM: best score resets.

## Risks and unknowns

- **Header fixup.** Tracked above. If it turns out the linker doesn't fix the checksum, we ship a 30-line script — not a blocker, just plumbing.
- **HDMA + STAT interaction on title screen.** GP-DMA halts the CPU; if STAT fires during a transfer, the palette swap is delayed. Mitigation: stage all title-screen HDMA *before* enabling the STAT interrupt; reverse order on transition out.
- **Double-speed entry on cold boot.** KEY1 prepare must be followed by `STOP` with interrupts disabled and joypad set up correctly per Pan Docs. Mitigation: a single canonical entry routine in `main.asm` reused everywhere; never re-enter.
- **Sample size growth.** If 4 banks turns out to be tight (font + tiles + screens crowd ROM2/ROM3), promote to 8 banks — same MBC5 controller, no other change. Linker config is the only thing that moves.
